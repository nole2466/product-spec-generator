# PM 角色定義

> Product Manager：定義需求和優先級

---

## 角色概述

| 項目 | 說明 |
|-----|------|
| **核心職責** | 定義「為什麼」和「要什麼」 |
| **主要產出** | PRD（產品需求文件）、Backlog |
| **協作對象** | PD、Stakeholders |
| **不負責** | 規格細節、技術決策、設計細節 |

## 輸入 / 輸出

### 輸入（你需要讀什麼）

| 來源 | 文件 | 目的 |
|-----|------|------|
| Stakeholders | 業務需求、用戶反饋 | 了解需求來源 |
| 數據分析 | 用戶行為、市場趨勢 | 驗證需求價值 |

## 職責邊界

### ✅ PM 該做的

- 定義使用者是誰
- 描述使用者的問題和需求
- 說明功能的商業價值
- 決定功能優先級
- 定義成功指標（KPI）
- 管理 Backlog

### ❌ PM 不該做的

- 決定技術實作方式
- 定義 API 格式
- 設計 UI/UX 細節
- 寫驗收測試的技術步驟

### 邊界範例

```markdown
✅ PM 說：
「用戶需要能搜尋股票，這樣可以快速找到想追蹤的標的」

❌ PM 不該說：
「用 Elasticsearch 實作搜尋，API 用 GET /api/search」
（這是 Backend 的決定）

❌ PM 不該說：
「搜尋框要放在右上角，用藍色按鈕」
（這是 PD 的決定）
```

## 需求撰寫指南

### 使用者故事格式

```markdown
作為 {角色}，
我想要 {功能}，
以便於 {價值}。
```

**好的範例**：
```markdown
作為 想追蹤股票的投資人，
我想要 用關鍵字搜尋股票，
以便於 快速找到想追蹤的標的。
```

**壞的範例**：
```markdown
作為 用戶，
我想要 搜尋功能，
以便於 搜尋。
（太模糊，沒有具體價值）
```

### 需求描述 Checklist

- [ ] 說明了使用者是誰
- [ ] 說明了使用者的問題
- [ ] 說明了功能的價值
- [ ] 定義了功能範圍（In/Out of Scope）
- [ ] 沒有包含技術實作細節
- [ ] 沒有包含 UI 設計細節

## KPI / 成功指標

每個功能都應該定義成功指標，用於衡量功能是否達成預期目標。

### 常見指標類型

#### 功能型 KPI

衡量功能本身的使用狀況：

| 指標 | 定義 | 計算方式 | 目標範例 |
|-----|------|---------|---------|
| 使用率 | 使用該功能的用戶比例 | 使用功能 DAU / 總 DAU | > 30% |
| 完成率 | 完成整個流程的比例 | 完成用戶 / 開始用戶 | > 80% |
| 錯誤率 | 遇到錯誤的用戶比例 | 遇錯用戶 / 使用用戶 | < 5% |
| 放棄率 | 中途放棄的用戶比例 | 放棄用戶 / 開始用戶 | < 20% |

#### 業務型 KPI

衡量功能對業務的貢獻：

| 指標 | 定義 | 計算方式 | 目標範例 |
|-----|------|---------|---------|
| 轉換率 | 完成目標行為的比例 | 轉換用戶 / 進入頁面用戶 | > 10% |
| 留存率 | N 日後回訪的比例 | N 日後活躍 / 首次使用 | D7 > 40% |
| 客單價 | 每用戶平均消費 | 總營收 / 付費用戶數 | 提升 10% |
| NPS | 淨推薦值 | 推薦者% - 貶損者% | > 50 |

#### 效能型 KPI

衡量功能的技術表現：

| 指標 | 定義 | 計算方式 | 目標範例 |
|-----|------|---------|---------|
| 回應時間 | API 回應時間 | P95 回應時間 | < 500ms |
| 可用性 | 服務正常運作比例 | Uptime / 總時間 | > 99.9% |
| 錯誤率 | 請求失敗比例 | 失敗請求 / 總請求 | < 0.1% |

### KPI 定義範本

在 spec.md 的 Requirements 區塊定義：

```markdown
### 成功指標

| 指標 | 目標 | 量測方式 | 基準線 |
|-----|------|---------|-------|
| 搜尋使用率 | > 50% | 使用搜尋 DAU / 總 DAU | 目前無此功能 |
| 搜尋完成率 | > 80% | 點擊結果 / 發起搜尋 | N/A |
| 搜尋回應時間 | < 500ms | P95 | N/A |

**量測週期**：上線後 2 週

**數據來源**：
- 用戶行為：Mixpanel
- 效能數據：Datadog
```

### KPI 選擇指南

| 功能類型 | 建議 KPI |
|---------|---------|
| 新功能 | 使用率、完成率 |
| 成長功能 | 轉換率、留存率 |
| 優化功能 | 完成率、錯誤率、回應時間 |
| 營收功能 | 轉換率、客單價 |

### 常見錯誤

❌ **沒有基準線**
```markdown
目標：提升轉換率
（提升多少？從多少提升？）
```

✅ **有基準線**
```markdown
目標：轉換率從 5% 提升至 8%
基準線：過去 30 天平均 5.2%
```

❌ **無法量測**
```markdown
目標：提升用戶體驗
（怎麼量測「體驗」？）
```

✅ **可量測**
```markdown
目標：NPS 從 45 提升至 55
量測方式：上線後發送 NPS 問卷
```

## Backlog 管理（Level 3）

### 目錄結構

```
backlog/
├── _index.md              # Dashboard
├── iterations/            # 迭代計畫
│   ├── v1.0.md
│   └── v1.1.md
├── features/              # 功能需求
│   ├── FE-001-search.md
│   └── FE-002-filter.md
├── bugs/                  # Bug
│   └── BUG-001-xxx.md
└── tech-debt/             # 技術債
    └── TD-001-xxx.md
```

### _index.md 範本

```markdown
# Backlog 總覽

## 當前迭代：v1.1

**目標**：{迭代目標}
**時程**：{開始} ~ {結束}

| 狀態 | ID | 名稱 | 類型 | 負責人 |
|:---:|-----|------|:----:|-------|
| 🔵 | FE-002 | 搜尋功能 | Feature | @pd |
| ⚪ | BUG-001 | 列表閃爍 | Bug | - |
| ✅ | FE-001 | 新增股票 | Feature | @rd |

## 統計

| 類型 | 總數 | 完成 | 進行中 | 待開始 |
|-----|:----:|:----:|:-----:|:-----:|
| Feature | 5 | 3 | 1 | 1 |
| Bug | 2 | 1 | 0 | 1 |
| Tech Debt | 1 | 0 | 0 | 1 |
```

## 審核 Checklist

當 PM 審核規格時，檢查以下項目：

### 需求完整性
- [ ] 需求描述清楚，無歧義
- [ ] 使用者故事符合格式
- [ ] 功能範圍明確（In/Out of Scope）
- [ ] 商業價值明確

### 可行性
- [ ] 需求可在合理時間內完成
- [ ] 沒有不合理的技術要求
- [ ] 沒有遺漏的依賴功能

### 驗收標準
- [ ] 每個需求都有對應的驗收標準
- [ ] 驗收標準可測試、可量化

## 待確認事項（Outstanding Items）

每份 PRD 都應該有「待確認事項」區塊，記錄所有未決定的項目：

```markdown
## 待確認事項

| # | 類型 | 項目 | 負責人 | 狀態 | 備註 |
|:-:|:---:|------|-------|:----:|------|
| 1 | 待確認 | 問卷頁面 UI 設計稿 | @PD | 🔵 | - |
| 2 | 相依性 | Design System | @PD | 🔵 | 延遲影響：無法開始開發 |
| 3 | 決策 | 是否支援訪客模式 | @PM | 🔵 | - |
```

### 類型說明

| 類型 | 說明 |
|-----|------|
| 待確認 | 需要確認的規格細節 |
| 相依性 | 依賴其他功能或資源 |
| 決策 | 需要做出的產品決策 |
| 問題 | 待釐清的問題 |

## Related Documents（相關文件）

每份 PRD 都應該連結到相關的 Spec、Contract、Acceptance：

```markdown
## Related Documents

| Type | Document | Owner |
```

## AI 協作指南

### 讓 AI 幫你寫需求

**Prompt 範本**：

```
我是 PM，需要建立一個功能的需求規格。

功能概述：
{簡述功能}

目標用戶：
{用戶描述}

請幫我產出 PRD，包含：
1. 背景與目標
2. 用戶與場景（User Story 格式）
3. 功能需求（Must Have / Should Have / Out of Scope）
4. 待確認事項

請不要包含技術實作細節和 UI 設計細節。
```

### 讓 AI 審核需求

**Prompt 範本**：

```
請以 PM 的角度審核以下 PRD：

{PRD 內容}

檢查項目：
1. 背景與目標是否清楚
2. 使用者故事是否完整
3. 功能範圍是否明確
4. 是否有遺漏的需求
5. 是否不小心包含了技術細節
6. 待確認事項是否完整

請產出：
- ✅ 通過項目
- ⚠️ 建議修改
- ❌ 必須修改
- ❓ 需要澄清
```
