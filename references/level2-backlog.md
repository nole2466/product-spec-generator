# Level 2 簡化版 Backlog

> 適合中型專案（1-2 週、3-5 人）的輕量任務追蹤

---

## 何時使用

| Level | 團隊規模 | 工時 | Backlog 類型 |
|:-----:|---------|------|-------------|
| 0-1 | 1-2 人 | < 5 天 | 不需要 |
| **2** | **3-5 人** | **1-2 週** | **簡化版** ✅ |
| 3 | > 5 人 | > 2 週 | 完整版 |

---

## 簡化版結構

```
specs/
└── search-stock/
    ├── spec.md
    ├── design.md
    ├── contract.md
    ├── acceptance.md
    └── TASKS.md        ← 簡化版 Backlog（單檔）
```

相較於 Level 3 的完整 `backlog/` 目錄結構，Level 2 只需要一個 `TASKS.md` 檔案。

## 使用流程

### 1. 建立 TASKS.md

在 spec 目錄下建立：

```bash
specs/search-stock/TASKS.md
```

### 2. 列出所有任務

根據專案規模，典型任務包含：

**規格階段**：
- [ ] 需求規格
- [ ] 設計規格
- [ ] API 契約
- [ ] 驗收標準

**開發階段**：
- [ ] API 開發
- [ ] 前端開發
- [ ] 單元測試

**驗收階段**：
- [ ] 整合測試
- [ ] Bug 修復
- [ ] 上線

### 3. 每日更新

- 更新任務狀態
- 記錄完成日期
- 追蹤問題

### 4. 完成後歸檔

功能上線後，保留 TASKS.md 作為紀錄。

---

## 與 Level 3 的差異

| 項目 | Level 2 簡化版 | Level 3 完整版 |
|-----|---------------|---------------|
| 檔案數 | 1 個（TASKS.md）| 多個目錄和檔案 |
| 任務追蹤 | 單一表格 | 獨立 feature/bug 檔案 |
| 迭代管理 | 無 | iterations/ 目錄 |
| Bug 追蹤 | 簡易表格 | 獨立 bug 檔案 |
| Dashboard | 內嵌在檔案中 | 獨立 _index.md |
| 適合 | 單一功能 | 多功能、長期專案 |

---

## 升級到 Level 3

當出現以下情況，建議升級：

- [ ] 功能數量 > 3 個
- [ ] 團隊人數 > 5 人
- [ ] 專案週期 > 2 週
- [ ] 需要跨迭代追蹤
- [ ] Bug 數量增多，需要獨立管理

**升級步驟**：

1. 建立 `backlog/` 目錄結構
2. 將 TASKS.md 內容遷移至對應檔案
3. 建立 `_index.md` Dashboard

## AI 協作指南

### 讓 AI 幫你建立 TASKS.md

**Prompt 範本**：

```
請根據以下規格，建立 TASKS.md 任務追蹤：

功能：{功能名稱}
團隊：{成員列表}
預估工時：{天數}

請產出包含：
1. 概覽資訊
2. 任務清單（規格/開發/驗收三階段）
3. 空的問題追蹤區塊
```

### 讓 AI 幫你更新進度

**Prompt 範本**：

```
請更新以下 TASKS.md 的進度：

{目前的 TASKS.md}

更新內容：
- 任務 5 完成
- 任務 6 開始
- 新增問題：{問題描述}

請產出更新後的 TASKS.md。
```
