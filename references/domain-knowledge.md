# 知識領域規範

> 定義產品所需的專業知識，確保團隊和 AI 對「領域知識」有一致理解

---

## 為什麼需要知識領域文件？

| 問題 | 解法 |
|-----|------|
| 團隊缺乏領域知識，做出外行決策 | 建立領域知識庫 |
| 新人需要長時間學習領域知識 | 結構化的學習材料 |
| AI 不懂專業領域，產出不專業 | 提供領域 Context |
| 產品決策缺乏專業依據 | 有據可查的知識來源 |
| 不同背景的人溝通困難 | 統一的知識框架 |

---

## 商業邏輯 vs 知識領域

| 項目 | 商業邏輯 | 知識領域 |
|-----|---------|---------|
| 內容 | **我們的**業務怎麼運作 | **這個領域**的專業知識 |
| 來源 | 內部決策、商業模式 | 產業知識、專業理論 |
| 變動性 | 隨業務調整 | 相對穩定 |
| 範例 | 我們的退款規則、會員制度 | 電商產業知識、物流知識 |

**簡單說**：
- 商業邏輯 = 「我們決定這樣做」
- 知識領域 = 「這個領域是這樣運作的」

## 文件結構

```
knowledge/
├── business/                    # 商業邏輯（見 business-logic.md）
│
└── domain/                      # 知識領域
    ├── _index.md               # 總覽和學習路徑
    ├── fundamentals/           # 基礎知識
    │   └── {topic}.md
    ├── advanced/               # 進階知識
    │   └── {topic}.md
    ├── regulations/            # 法規知識
    │   └── {topic}.md
    ├── industry/               # 產業知識
    │   └── {topic}.md
    └── references/             # 外部參考
        └── {topic}.md
```

## 知識文件撰寫原則

### 1. 結構清晰

```markdown
# {主題}

## 概述
（這是什麼、為什麼重要）

## 核心概念
（關鍵術語和定義）

## 詳細說明
（深入解釋）

## 產品設計啟發
（這些知識如何影響我們的產品）

## 常見誤解
（容易搞錯的地方）

## 延伸閱讀
（進一步學習的資源）
```

### 2. 與產品關聯

每個知識點都要回答：**這對我們的產品有什麼影響？**

```markdown
## 產品設計啟發

| 知識點 | 產品影響 | 設計建議 |
|-------|---------|---------|
| 用戶在 X 情況下會 Y | 功能 A 需要考慮 | 建議做 Z |
```

### 3. 適當深度

- **基礎知識**：所有人都該懂的
- **進階知識**：需要時再查
- **參考資料**：連結外部資源

## 常見知識領域類型

### 1. 產業知識

了解所處產業的運作方式。

| 產業 | 常見知識主題 |
|-----|-------------|
| 電商 | 購物流程、物流、金流、客服 |
| 金融 | 監管法規、風控、合規 |
| 醫療 | 醫療法規、隱私保護、用藥知識 |
| 教育 | 學習理論、課程設計、評量方式 |
| 餐飲 | 食安法規、出餐流程、庫存管理 |

### 2. 技術知識

產品相關的技術背景知識。

| 領域 | 常見知識主題 |
|-----|-------------|
| 資料 | 資料格式、來源、更新頻率 |
| 地圖 | 座標系統、地理編碼、路徑規劃 |
| 支付 | 金流串接、安全規範、對帳 |
| 通知 | 推播機制、到達率、最佳實踐 |

### 3. 法規知識

必須遵守的法律規範。

| 類型 | 常見主題 |
|-----|---------|
| 隱私 | GDPR、個資法、Cookie 政策 |
| 消費者保護 | 退款政策、定型化契約 |
| 產業法規 | 依產業而定 |
| 智財權 | 商標、著作權、專利 |

### 4. 用戶知識

了解目標用戶的特性。

| 類型 | 常見主題 |
|-----|---------|
| 用戶研究 | Persona、用戶旅程、痛點 |
| 行為模式 | 使用習慣、決策流程 |
| 心理學 | 認知負荷、習慣養成 |

## AI 協作指南

### 讓 AI 具備領域知識

**Prompt 範本**：

```
你是一個 {領域} 產品的設計助手。請先閱讀以下領域知識：

基礎知識：
{fundamentals 內容}

法規要求：
{regulations 內容}

現在請幫我設計「{功能名稱}」的規格，
確保符合領域專業和法規要求。
```

### 讓 AI 幫你整理領域知識

**Prompt 範本**：

```
請幫我將以下領域知識整理成結構化文件：

{零散的知識或資料}

請整理成：
1. 核心概念（表格形式）
2. 運作方式（流程或說明）
3. 產品設計啟發
4. 常見誤解
```

### 讓 AI 檢查領域正確性

**Prompt 範本**：

```
請檢查以下規格的領域知識是否正確：

{spec.md 內容}

參考知識：
{相關 domain 文件}

檢查：
1. 術語使用是否正確
2. 流程是否符合領域慣例
3. 是否有常識性錯誤
4. 是否符合法規要求
```
